# Design: 请求数据采集与训练数据导出

Generated by /office-hours on 2026-05-29
Branch: dev
Status: DRAFT

## Problem Statement

LumenRoute 作为所有模型请求的中心代理，当前只记录请求元数据（路由、延迟、token 数等），不存储 request/response body。团队需要将通过 LumenRoute 代理的请求数据作为训练数据集，导出到评测项目 ipsa-eval。不同项目组使用不同的数据类别（如 RAG、QA），需要按项目维度组织和过滤数据。

## Constraints

1. **规模**: 未来 ~7000 用户，请求量大，body 存储不能放 SQLite
2. **性能**: 采集不能影响代理延迟，必须异步写入
3. **兼容性**: ipsa-eval 支持 prompt 和 rag 两种 schema，导入格式为 JSONL
4. **可选性**: 不是所有路由都需要采集，需要按项目开关控制
5. **存储分离**: body 数据写入文件系统，元数据留在 SQLite
6. **内网场景**: 很多请求不经过 API Key 认证，不能用 key 区分项目

## Premises

1. 存储完整的 request body 和 response body（流式拼接后的完整响应）
2. 新增「项目」(Project) 概念来组织数据，路由关联到项目
3. body 数据写入文件系统（JSONL），元数据留在 SQLite
4. 导出 API 支持按项目、时间、模型等维度过滤，输出 JSONL
5. 流式请求在代理层拼接完整响应后再存储
6. 采集功能 opt-in，按项目/路由维度控制

## 0. 采集策略

### 0.1 什么会被采集

| 场景 | 是否采集 | 说明 |
|------|---------|------|
| 路由未关联项目 | 否 | `route.project_id IS NULL` 时跳过 |
| 全局开关关闭 | 否 | `LUMENROUTE_CAPTURE_ENABLED=false` 时全部跳过 |
| 项目采集关闭 | 否 | `project.capture_enabled=false` 时跳过 |
| 非流式成功请求 | 是 | 记录完整 request + response body |
| 非流式失败请求（4xx/5xx） | 是 | 记录 request body + 错误 response body |
| 流式完整响应 | 是 | 拼接后记录完整响应 |
| 流式未完成（upstream EOF） | 是 | 记录已收到的部分内容，标记 `stream_completed: false` |
| 上游连接失败 | 否 | 无 response body，只有元数据（已在 request_logs 中） |
| 采样命中 | 按概率 | `rand.Float64() < project.sample_rate` |

### 0.2 开关优先级

```
全局 LUMENROUTE_CAPTURE_ENABLED == false  → 全部跳过（最高优先级）
route.project_id == nil                    → 跳过
project.capture_enabled == false           → 跳过
project.sample_rate < rand()               → 跳过（采样未命中）
其他                                       → 采集
```

### 0.3 request_id 一致性

`request_id` 在 `ChatCompletions` 入口处生成一次，同时传递给 `writeLog` 和 `maybeCapture`。这样 `request_logs` 和 `request_captures` 通过 `request_id` 精确关联，可以从采集记录追溯到完整的请求元数据。

```go
func (s *Service) ChatCompletions(w http.ResponseWriter, r *http.Request) {
    // 在入口生成一次，后续共享
    requestID := logs.GenerateRequestID("")
    // ...
}
```

### 0.4 Body 大小保护

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `LUMENROUTE_CAPTURE_MAX_BODY_SIZE` | 1MB (1048576) | 单个 request 或 response body 的最大采集大小 |

超过限制时**不采集该条记录**（整条跳过），在 `request_captures` 元数据中标记 `body_skipped=true`。判断发生在 Submit 之前（proxy 层），不进入 channel，避免大 body 占满内存缓冲。不截断 JSON body 是因为截断后的 JSON 无法解析，对下游消费者来说是垃圾数据。

### 2.3 流式响应重建

流式请求的响应从 SSE chunks 拼接为标准 chat completion 格式：

```go
type ReconstructedResponse struct {
    ID      string `json:"id"`
    Object  string `json:"object"`
    Model   string `json:"model"`
    Choices []struct {
        Message      ChatMessage `json:"message"`
        FinishReason string      `json:"finish_reason"`
    } `json:"choices"`
    Usage *TokenUsage `json:"usage,omitempty"`
}
```

数据来源（与 §4.5 实现一致）：
- `id`: 从第一个 SSE chunk 解析；不可用时生成 `chatcmpl-{random}`
- `model`: 优先从第一个 chunk 获取，fallback 从 request body 读取
- `choices[0].message.content`: 从所有 `delta.content` 拼接（现有 `accumulated` 变量）
- `finish_reason`: 从最后一个含 `finish_reason` 的 chunk 获取，默认 `"stop"`
- `usage`: 优先从最后一个 chunk 的 `usage` 字段获取（部分上游支持），fallback 用 tiktoken 估算

`time_to_first_chunk_ms` 等同于"首 token 用时"（TTFT），与 `request_logs` 中的字段名一致。

这样下游消费者不需要关心请求是流式还是非流式，拿到的都是相同结构。

**已知局限（v1 scope）**: 只采集 `choices[0].delta.content` 文本。Tool calls、reasoning blocks、多 choice 等结构不在 v1 采集范围内。如需扩展，需改造 `SSEScanner` 以累积结构化 choice 状态。

## 3. Capture Service 架构

新增 `internal/capture/` 包。

```
internal/capture/
  ├── types.go        # CaptureEntry, CaptureRecord, Config
  ├── service.go      # Service（异步写入调度）
  ├── writer.go       # FileWriter（JSONL 文件写入）
  ├── store.go        # Store 接口 + SQLite 实现（元数据）
  └── cleanup.go      # 按项目的保留策略清理
```

### 3.1 异步写入流程

```
Proxy Handler
    │
    ├── 代理请求正常处理（不阻塞）
    │
    └── captureService.Submit(entry)
            │
            ▼
        buffered channel (size: 256)
            │
            ▼
        Writer goroutine
            │
            ├── 写入 JSONL 文件（append mode）
            ├── batch fsync（每批/每秒）
            └── batch INSERT CaptureRecord 到 SQLite（每 100 条/每秒）
```

关键设计：
- **channel buffer 256**: 防止瞬时峰值阻塞代理，限制内存（每条 entry 含完整 body）
- **channel 满时丢弃**: 采集不能影响代理可用性，丢弃时记录 metrics
- **单写入 goroutine**: 避免文件写入竞争
- **文件 append**: 单写入者，天然无竞争
- **batch fsync**: 每批写入结束或每秒调用一次 fsync，平衡持久性和性能
- **写入顺序**: 文件写入在 SQLite 写入之前，确保元数据引用的文件内容已落盘
- **batch INSERT**: 每 100 条或每秒一次批量写入 SQLite，减少与 logs.Service 的写入竞争
- **错误处理**: 文件写入失败记 `capture_write_errors_total` + error log；SQLite 写入失败记 `capture_store_errors_total` + error log

### 3.2 Service 核心

```go
type Service struct {
    entries    chan CaptureEntry
    writer     *FileWriter
    store      Store
    dropped    atomic.Int64
    basePath   string
    wg         sync.WaitGroup
}

type CaptureEntry struct {
    RequestID         string
    ProjectID         int64
    RouteName         string
    PublicModelName   string
    UpstreamModelName string
    ProviderName      string
    Stream            bool
    StatusCode        int
    LatencyMs         int
    TTFCMs            *int
    PromptTokens      *int
    CompletionTokens  *int
    RequestBody       json.RawMessage
    ResponseBody      json.RawMessage
}

func (s *Service) Submit(entry CaptureEntry) {
    select {
    case s.entries <- entry:
    default:
        s.dropped.Add(1)
    }
}

func (s *Service) Start() {
    s.wg.Add(1)
    go s.processLoop()
}

func (s *Service) processLoop() {
    defer s.wg.Done()
    batch := make([]CaptureRecord, 0, 100)
    ticker := time.NewTicker(1 * time.Second)
    defer ticker.Stop()

    flush := func() {
        if len(batch) == 0 {
            return
        }
        s.writer.Sync() // batch fsync
        if err := s.store.InsertBatch(batch); err != nil {
            captureStoreErrorsTotal.Add(int64(len(batch)))
            log.Printf("capture store batch error: %v", err)
        }
        batch = batch[:0]
    }

    for {
        select {
        case entry, ok := <-s.entries:
            if !ok {
                flush()
                return
            }
            filePath, offset, err := s.writer.Write(entry)
            if err != nil {
                captureWriteErrorsTotal.Add(1)
                log.Printf("capture write error: %v", err)
                continue
            }
            batch = append(batch, CaptureRecord{
                RequestID:       entry.RequestID,
                ProjectID:       entry.ProjectID,
                PublicModelName: entry.PublicModelName,
                Stream:          entry.Stream,
                StatusCode:      entry.StatusCode,
                BodySkipped:     entry.BodySkipped,
                FilePath:        filePath,
                FileOffset:      offset,
                RequestSize:     len(entry.RequestBody),
                ResponseSize:    len(entry.ResponseBody),
            })
            if len(batch) >= 100 {
                flush()
            }
        case <-ticker.C:
            flush()
        }
    }
}

func (s *Service) Close() {
    close(s.entries)
    s.wg.Wait()
    s.writer.Close() // 关闭 LRU 文件池中所有句柄
}
```

### 3.3 FileWriter（LRU 文件句柄池）

多项目场景下，单文件句柄会因频繁 open/close 产生 thrashing。使用 LRU 池管理活跃的文件句柄：

```go
type FileWriter struct {
    basePath string
    pool     *lruFilePool // 容量 16-32，按 LRU 淘汰关闭
}

func (w *FileWriter) Write(entry CaptureEntry) (string, int64, error) {
    targetPath := w.buildPath(entry.ProjectID)
    f, err := w.pool.Get(targetPath) // LRU hit → 直接用；miss → open + 淘汰最旧
    if err != nil {
        return "", 0, err
    }

    offset, _ := f.Seek(0, io.SeekEnd)

    line := marshalCaptureLine(entry)
    f.Write(line)
    f.Write([]byte("\n"))

    return targetPath, offset, nil
}

func (w *FileWriter) Sync() {
    w.pool.SyncAll() // 对所有打开的文件调用 fsync
}

func (w *FileWriter) Close() {
    w.pool.CloseAll()
}

func (w *FileWriter) buildPath(projectID int64) string {
    now := time.Now()
    dir := filepath.Join(w.basePath,
        fmt.Sprintf("%d", projectID),
        now.Format("2006-01-02"))
    os.MkdirAll(dir, 0755)
    return filepath.Join(dir, now.Format("15")+".jsonl")
}
```

### 4.4 maybeCapture 方法（位于 proxy/capture.go）

调用者在 `proxyNonStream`/`proxyStream` 中构造 `CaptureEntry` struct，传递给 `maybeCapture`，避免 11 个独立参数：

```go
func (s *Service) maybeCapture(entry capture.CaptureEntry) {
    if s.captureService == nil {
        return
    }
    if entry.ProjectID == 0 || !s.captureEnabled {
        return
    }

    proj := s.getProjectCached(entry.ProjectID)
    if proj == nil || !shouldCapture(proj) {
        return
    }

    // Body 大小保护：超限时整条跳过（不截断，避免产生无法解析的 JSON）
    maxSize := s.captureMaxBodySize
    if len(entry.RequestBody) > maxSize || len(entry.ResponseBody) > maxSize {
        entry.RequestBody = nil
        entry.ResponseBody = nil
        entry.BodySkipped = true
    }

    s.captureService.Submit(entry)
}
```

### 4.4.1 项目配置缓存

在代理热路径中每次请求都查 SQLite 获取项目配置是不可接受的。使用带 TTL 的内存缓存：

```go
type projectCache struct {
    mu      sync.RWMutex
    items   map[int64]*cachedProject
    ttl     time.Duration  // 默认 60 秒
}

type cachedProject struct {
    project   *project.Project
    expiresAt time.Time
}

func (s *Service) getProjectCached(projectID int64) *project.Project {
    s.projCache.mu.RLock()
    if cached, ok := s.projCache.items[projectID]; ok {
        if time.Now().Before(cached.expiresAt) {
            s.projCache.mu.RUnlock()
            return cached.project
        }
    }
    s.projCache.mu.RUnlock()

    proj, err := s.projectService.Get(projectID)
    if err != nil {
        return nil
    }

    s.projCache.mu.Lock()
    s.projCache.items[projectID] = &cachedProject{
        project:   proj,
        expiresAt: time.Now().Add(s.projCache.ttl),
    }
    s.projCache.mu.Unlock()
    return proj
}
```

项目配置变更后最多 60 秒生效。Admin API 更新项目时可主动清除缓存。

### 4.5 流式响应重建函数

新增 `internal/proxy/reconstruct.go`：

```go
func reconstructStreamResponse(completionText string,
    sr *StreamResult, scanner *SSEScanner, reqBody []byte) []byte {

    // 优先从 SSE chunk 提取 id 和 model
    chunkID := scanner.FirstChunkID()
    chunkModel := scanner.FirstChunkModel()

    var req map[string]interface{}
    json.Unmarshal(reqBody, &req)

    if chunkID == "" {
        chunkID = fmt.Sprintf("chatcmpl-%x", rand.Uint64())
    }
    if chunkModel == "" {
        chunkModel, _ = req["model"].(string)
    }

    finishReason := scanner.LastFinishReason()
    if finishReason == "" {
        finishReason = "stop"
    }

    resp := map[string]interface{}{
        "id":      chunkID,
        "object":  "chat.completion",
        "model":   chunkModel,
        "choices": []map[string]interface{}{{
            "index": 0,
            "message": map[string]string{
                "role":    "assistant",
                "content": completionText,
            },
            "finish_reason": finishReason,
        }},
    }

    if sr.TotalTokens > 0 {
        resp["usage"] = map[string]int{
            "prompt_tokens":     sr.PromptTokens,
            "completion_tokens": sr.CompletionTokens,
            "total_tokens":      sr.TotalTokens,
        }
    }

    out, _ := json.Marshal(resp)
    return out
}
```

需要扩展 `SSEScanner`，新增 `FirstChunkID()`、`FirstChunkModel()`、`LastFinishReason()` 方法，从 SSE 流中提取这些字段。

### 6.3 导出 Handler 实现要点

```go
func (h *CaptureHandler) Export(w http.ResponseWriter, r *http.Request) {
    projectID := parseID(r)
    filter := parseCaptureFilter(r)

    w.Header().Set("Content-Type", "application/x-ndjson")
    w.Header().Set("Transfer-Encoding", "chunked")

    flusher, _ := w.(http.Flusher)

    // 游标分页：从 cursor 位置开始，每次取 page_size 条
    // SQL: WHERE id > ? AND project_id = ? AND ... ORDER BY id LIMIT ?
    records, _ := h.store.ListAfterCursor(projectID, filter.Cursor,
        filter.PageSize, filter)

    for _, rec := range records {
        line, err := h.readCaptureLine(rec.FilePath, rec.FileOffset)
        if err != nil {
            continue
        }
        if filter.Format == "openai" {
            converted, ok := convertToOpenAIFormat(line)
            if !ok {
                captureExportSkippedTotal.Add(1)
                continue
            }
            out, _ := json.Marshal(converted)
            w.Write(out)
        } else {
            w.Write(line) // 原始 JSONL 字节，不经过 json.Encoder 避免双重编码
        }
        w.Write([]byte("\n"))
    }

    // 分页元数据嵌入 JSONL 流的最后一行（不用 HTTP header，避免 chunked 下 header 顺序问题）
    if len(records) > 0 {
        lastID := records[len(records)-1].ID
        hasMore := len(records) == filter.PageSize
        meta, _ := json.Marshal(map[string]interface{}{
            "_meta": map[string]interface{}{
                "next_cursor": lastID,
                "has_more":    hasMore,
            },
        })
        w.Write(meta)
        w.Write([]byte("\n"))
    }
    flusher.Flush()
}
```

`readCaptureLine` 实现：从文件的 `offset` 位置 Seek，读取到下一个 `\n` 为止，得到完整 JSON 行。遇到损坏或截断行时跳过并记录 metric。

**format=openai 转换规则**:

```go
func convertToOpenAIFormat(captureLine []byte) (map[string]interface{}, bool) {
    var data map[string]interface{}
    if err := json.Unmarshal(captureLine, &data); err != nil {
        return nil, false
    }

    reqBody, ok := data["request_body"].(map[string]interface{})
    if !ok {
        return nil, false
    }
    respBody, ok := data["response_body"].(map[string]interface{})
    if !ok {
        return nil, false
    }

    messages, ok := reqBody["messages"].([]interface{})
    if !ok {
        return nil, false
    }
    choices, ok := respBody["choices"].([]interface{})
    if !ok || len(choices) == 0 {
        return nil, false
    }
    choice0, ok := choices[0].(map[string]interface{})
    if !ok {
        return nil, false
    }
    assistantMsg, ok := choice0["message"]
    if !ok {
        return nil, false
    }

    allMessages := append(messages, assistantMsg)
    return map[string]interface{}{"messages": allMessages}, true
}
```

### 6.4 导出认证方案

导出 API 需要支持两种认证场景：

| 场景 | 认证方式 | 说明 |
|------|---------|------|
| Admin 手动导出 | Session cookie | 与现有 Admin API 一致 |
| ipsa-eval 自动拉取 | Project Export Token | 专用 token，作用域限定到单个项目 |

**Project Export Token**:

- 项目创建时可选生成，也可后续通过 API 生成/轮换
- 格式: `lrx_{project_id}_{random_32chars}`
- 存储在 `projects` 表的 `export_token_hash` 列（bcrypt hash）
- 请求时通过 `Authorization: Bearer lrx_...` 传递
- 只能访问该项目的导出 API（captures/export），不能操作其他资源

```go
// 导出 API 中间件
func (h *CaptureHandler) authExport(r *http.Request, projectID int64) error {
    // 1. 先检查 session（Admin 用户）
    if h.authService.HasValidSession(r) {
        return nil
    }
    // 2. 检查 Export Token
    token := extractBearerToken(r)
    if token == "" {
        return ErrUnauthorized
    }
    return h.projectService.ValidateExportToken(projectID, token)
}
```

**ipsa-eval 字段映射表**:

## 7. 清理策略

扩展 `internal/scheduler/` 包，新增 `capture.go`：

清理分两步，以 SQLite 数据为驱动（而非目录扫描），确保索引和文件一致：

```go
func StartCaptureCleanup(projectSvc *project.Service,
    captureStore capture.Store, basePath string) {

    ticker := time.NewTicker(1 * time.Hour)
    for range ticker.C {
        projects, _ := projectSvc.ListAll()
        for _, proj := range projects {
            if proj.RetentionDays <= 0 {
                continue
            }
            cutoff := time.Now().AddDate(0, 0, -proj.RetentionDays)
            cleanupProject(captureStore, basePath, proj.ID, cutoff)
        }
    }
}

func cleanupProject(store capture.Store, basePath string,
    projectID int64, cutoff time.Time) {

    // Step 1: 查询过期记录涉及的文件路径（去重）
    expiredFiles, err := store.ListExpiredFiles(projectID, cutoff)
    if err != nil {
        captureCleanupErrorsTotal.Add(1)
        log.Printf("cleanup: list expired files failed for project %d: %v", projectID, err)
        return
    }

    // Step 2: 删除 SQLite 中的过期记录
    if err := store.DeleteBefore(projectID, cutoff); err != nil {
        captureCleanupErrorsTotal.Add(1)
        log.Printf("cleanup: delete records failed for project %d: %v", projectID, err)
        return
    }

    // Step 3: 对每个文件，检查是否还有存活记录引用它
    for _, filePath := range expiredFiles {
        remaining, err := store.CountByFile(filePath)
        if err != nil {
            captureCleanupErrorsTotal.Add(1)
            log.Printf("cleanup: count by file failed for %s: %v", filePath, err)
            continue
        }
        if remaining == 0 {
            if err := os.Remove(filepath.Join(basePath, filePath)); err != nil && !os.IsNotExist(err) {
                captureCleanupErrorsTotal.Add(1)
                log.Printf("cleanup: remove file failed for %s: %v", filePath, err)
            }
        }
    }

    // Step 4: 清理空的日期目录
    projDir := filepath.Join(basePath, fmt.Sprintf("%d", projectID))
    entries, _ := os.ReadDir(projDir)
    for _, e := range entries {
        if !e.IsDir() {
            continue
        }
        subEntries, _ := os.ReadDir(filepath.Join(projDir, e.Name()))
        if len(subEntries) == 0 {
            os.Remove(filepath.Join(projDir, e.Name()))
        }
    }
}
```

关键改进：只有当一个文件不再被任何存活的 `request_captures` 行引用时才删除，避免索引/文件不一致。

## 14. Success Criteria

## 19. Failure Modes

| 故障场景 | 用户影响 | 测试覆盖 | 错误处理 |
|---------|---------|---------|---------|
| JSONL 文件写入失败（磁盘满） | 采集数据丢失，代理正常 | capture/service_test.go | metric + log，继续处理下一条 |
| SQLite batch INSERT 失败 | 元数据丢失，文件数据孤立 | capture/store_test.go | metric + log，batch 级重试 |
| Channel 满载丢弃 | 采集不完整，代理正常 | capture/service_test.go | dropped counter + log |
| readCaptureLine 文件损坏 | 导出时跳过该条 | capture_handlers_test.go | skip + metric |
| 项目缓存过期 | 配置变更最多 60s 延迟生效 | proxy/capture_test.go | TTL 过期后重新加载 |
| 清理时文件被 export 读取 | 读取可能失败 | **critical gap** | 需要文件级协调或容错 |
| Graceful shutdown 中断 | Close() drain 确保不丢 | capture/service_test.go | channel drain + final flush |
| Export token 泄露 | 项目数据泄露 | capture_handlers_test.go | 项目级作用域限制爆炸半径 |

**Critical gap**: 清理过程删除文件时，export handler 可能正在读取同一文件。需要在导出路径增加容错（文件不存在时 skip + metric），或对清理操作增加文件级锁。建议导出路径增加 `os.IsNotExist` 检查作为最小修复。
